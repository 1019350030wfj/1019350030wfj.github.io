AAPT

AAPT全称，Android Asset Packaging Tool。在Android SDK的build－tools目录下找到。 它可以查看、创建、修改压缩文件（zip包，jar包，apk文件），也可以将资源编译成二进制文件。实际上不管是Android Studio还是Eclipse都是调用AAPT，将所有文件打包成一个可以运行的APK文件。

使用方法

1、列出apk中的所有内容

aapt list apk路径 ［ > 将内容保存到apk。txt］
后面还可以加 －v参数 可以以表格的方式展示查询到的内容
－a参数  可以输入所有目录下的内容

2、查看指定文件信息

aapt d[ump] [--values] 《apk path》

   badging          Print the label and icon for the app     declared in APK.

   permissions      Print the permissions from the APK.

   resources        Print the resource table from the APK.

   configurations   Print the configurations in the APK.

   xmltree          Print the compiled xmls in the given assets.

   xmlstrings       Print the strings of the given compiled xml asse

3、修改apk包

aapt pachage  打包生成资源压缩包

aapt remove 从压缩包中删除指定文件

appt add 向压缩包中添加指定文件

4、显示aapt版本信息

aapt version apk路径

UI性能分析

UI性能的目标：

- 减少绘图的等待时间
- 使帧率更加平稳、连贯

16ms 黄金准则：

一般来说，Android设备的屏幕刷新率为60帧每秒。要保持画面流畅不卡顿，就需要让每一帧的时间不超过1/60fps ＝ 16。6ms每帧。 如果中间某些帧的渲染时间超过16ms，就会导致这段时间的画面发生跳帧，因此原本流畅的画面便会发生卡顿。

Android 系统对UI的提升

Android系统对UI性能的优化工作始终没有停止过。
－ 软解时代：在android2.3时代，所有绘制由CPU完成，即通过软件运算画图。
－ 硬解时代：在Android2.3之后，Android系统增加了GPU，同时把很多绘图工作交给GPU进行渲染。
－ 黄油时代：在Android4.1之后，Google发布了“Project Butter”－－黄油计划。通过VSYNC垂直同步机制和多缓冲机制（three frame buffer）进一步提高绘制效率。
－ 异步绘制时代：在Android5.0之后，系统增加了Render Thread。通过这个线程进行异步绘制，即使某一帧发生延迟也不会影响下一帧的绘制。

RelativeLayout VS LinearLayout

LinearLayout通常需要进行嵌套使用，这样在布局深度上RelativeLayout更有优势。
但是RelativeLayout大部分时间需要进行多次测量，才能确定View的大小。LinearLayout只有在使用weight属性后，才回发生两次测量。
使用LinearLayout一定要保证层级不要太深；使用RelativeLayout，则需要尽量避免嵌套。

工具：

Hierachy Viewer。 从这个工具中可以看出，如果是线型的View树，则可以用后面的View替代它的父View。避免冗余的布局来提高性能。

内存探究：

内存区分：
1、寄存器Registers：用于存储指令、地址、数据。
2、栈Stack：存储基本数据类型，对象的引用和函数地址等，由系统控制
3、堆Heap：存放对象本身和数组，由开发者控制
4、静态域static field：存储静态变量
5、常量池constant：存储常量

AcitivityMananger.MemoryInfo:
- totalMem:系统可用总内存
- availMem:系统当前可用内存（剩余内存）
- lowMemory:是否处于低内存状态
- thrshold:内存阈值

Runtime是一个非常有用的类，可用获取很多运行时的数据。
－ totalMemory：VM Heap Size
－ freeMemory：Free VM Heap Size
－ maxMemory：VM Heap Size Limit


获取更多内存：
1、通过子进程：通过android：process指定新进程后，即可申请新的内存分配。webview 通常运行在单独的子进程中。通过Kill Process的方式来完全回收内存。

2、通过使用Native Heap： Android系统限制的事Java Heap的内存大小，而Native Heap的内存分配是由系统控制的，大小不受限制。因此通过jni就可以实现对Native Heap的申请。

3、使用OpenGL：OpenGL的很多API不受Java Heap限制的，因为OpenGL将RAM的一部分作为显存使用。

4、LargeHeap：在AndroidManifest文件中，可用通过设置LargeHeap参数强制给APP分配更大的内存空间，适用于在运行时需要处理大量数据、图片等内容的APP。但该方法类似于“七伤拳”，在获取更大内存的同时也加大了GC的难度，使GC速度变慢而去在应用推出到后台时，也更容易被系统回收。

内存泄漏检测：

通过Dump Heap分析，查看Data object 的Total Size 的大小释放由很大变化。有，可能发生了内存泄漏，导致内存使用不断上升。 

Allocation Tracker：
Dump Heap工具帮助开发者对内存的整体使用情况进行掌控，但缺点时无法了解每块内存具体分配给哪些对象了。 这时就需要使用Allocation Tracker工具进行内存跟踪。

内存泄漏分析：

MAT（Memory Analysis Tools）工具是一个检测内存泄漏的利器。准备Dump Heap文件，通过AS或者DDMS生成的hprof文件都不是标准的格式，开发者需要通过sdk提供的工具－－hprof－conv进行转换，执行以下指令即可。
1、hprof－conv dump.hprof converted-dump.hprof

2、AS也可以对dump.hprof文件鼠标右键选择Export to standard hprof直接转换

MAT提供的数据非常多，但有两个功能是很常用，就是它的Histogram 和 DominatorTree功能。Histogram可以统计内存中对象的名称、种类、实例数和大小，而DominatorTree 则是建立这些内存对象之间的联系。

在Histogram界面，可以在class name区域输入过滤条件，比如输入Activity，就可以看到所有Activity类。假如说MainActivity的实例有17次之多，基本可以断定是内存泄漏了。 那么具体是如何泄漏的呢？可以通过查看GC对象的引用链来进行分析。在MainActivity上单击鼠标右键，选择Merge Shortest paths to GC Roots，并通过弹出的列表选择相关类型的引用（强、软、虚、弱），分析不同引用类型下的GC情况，这里选择With all references。可以看出MainActivity的引用被谁持有。
同时，在Histogram中还可以查看一个对象到底包含了哪些对象的引用。例如，要查看MainActivity所包含的引用，直接单击鼠标右键，选择list objects－－with incoming references（显示选中对象被哪些外部对象引用，而outcoming 是指选中对象持有 哪些对象的引用）

DominatorTree更侧重于关系的分析，而Histogram更侧重于量的分析。打开DominatorTree界面，同样可以在第一行输入过滤条件。
－ ShallowHeap：对象自身占用的内存大小，不包括它引用的对象
－ RetainedHeap：对象自身占用的内存大小，加上它直接或者间接引用对象的大小。开发者通过RetainedHeap来调查对象所占内存大小，RetainedHeap内存占用大的对象，应该被列为重点怀疑对象。
文件图标： 每个文件图标都代表一个对象，并占据图表一行。但仔细看，实际上这些文件图标是不一样的。有些文件右下角会有一个小圆点，如果有这个小圆点，则代表该对象可以被GC系统访问到。也就是说，带小圆点的对象都有可能是内存泄漏的可疑对象。 通常情况下，开发者首先需要根据RetainedHeap大小来确定嫌疑犯，再通过Path To GC Roots找到GC引用链调查它是否真的有嫌疑。

录制Gif

MP4转Gif：从Android4.4系统开始，ADB新增了一个screenrecord指令，在终端执行如下指令，即可开始对手机界面进行录制。
adb shell screenrecord ／sdcard／video.mp4
通过这句指令，系统会在180s内对手机进行录屏操作。当然也可以指定录制的时间喝帧率。

／／指定视频分辨率大小
adb shell screenrecord －－size 1280*720 ／sdcard／video.mp4

／／指定录制时间（示例为200s）
adb shell screenrecord －－time－limit 200 ／sdcard/video.mp4

／／指定录制比特率（示例为5Mbps，系统默认为4Mbps）
adb shell screenrecord －－bit－rate 5000000 ／sdcard/video.mp4

有了MP4文件，导出后可以用各种格式转换工具将MP4转换为Gif图。
